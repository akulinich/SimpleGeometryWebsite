# Песочница (Sandbox) — план

3D/2D геометрический редактор, работающий целиком в браузере без backend.
Отдельный React+Vite SPA, смонтированный на роуте `/sandbox`; кнопка «Песочница»
рядом со «Статьями» — просто ссылка на этот роут. Astro остаётся статиком.

## Стек

React + TypeScript + Vite + three.js + Zustand + C++/Wasm (Emscripten) + Web Worker.

## Главный принцип

**TypeScript управляет редактором, three.js рисует, C++ считает геометрию.**

- **Canonical model** сцены хранится отдельно от three.js-объектов — модель это
  истина, three.js это вид.
- C++-геометрия — **своя реализация** (это часть фишки проекта), не сторонние либы.
- 2D — это та же сцена, камера смотрит на плоскость XY.

---

## Шаг 1. Редактор end-to-end на чистом TS-геометрии

Цель: полноценный работающий редактор **без Wasm**. Сначала обкатываем сам
редактор, не смешивая его с новой wasm-трубой.

Объём:
- viewport + camera controls;
- примитивы;
- selection (picking — **в TS**, штатный three.js raycaster);
- transform (move/rotate/scale);
- import/export через JSON;
- переключение 2D ↔ 3D (камера на плоскость XY);
- undo/redo.

Ключевые решения:
- **Canonical model отдельно от three.js.** three.js-объекты пересобираются из
  модели; модель — единственный источник правды.
- **Undo/redo закладываем с первого дня.** Zustand + immer с inverse-патчами; все
  мутации модели идут через единый слой команд.
- **Per-frame состояние (драг гизмо, орбита камеры) мимо React.** Императивно
  дёргаем three.js и модель, в Zustand коммитим только на `dragend` — иначе шторм
  ре-рендеров.

## Шаг 2. Подключаем C++/Wasm-трубу

Цель: ввести границу JS↔Wasm и прогнать через неё полный круг на простом
алгоритме. **Первый мигрированный алгоритм — построение выпуклой оболочки 2D**
(как smoke-тест трубы, не потому что ему нужен C++).

Объём:
- C++ собирается в Wasm через Emscripten, сборка интегрирована в Vite;
- тяжёлые вычисления запускаются в **Web Worker** (single-threaded Wasm, **без
  pthreads** на MVP);
- convex hull считается в C++, вызывается из редактора, результат рисуется.

Ключевые решения:
- **Взаимодействие batch-style:** передаём большие typed arrays, а не много мелких
  вызовов.
- **Геометрию пока копируем через границу на каждый вызов** (без хранения буферов
  wasm-side и хэндлов — это возможная оптимизация на потом).
- **Один marshalling-модуль владеет всеми переходами границы.** Остальной код
  видит только чистые типизированные функции (`convexHull(Float32Array):
  Float32Array`) и про wasm не знает.
- Голый `extern "C"` + ручной `malloc`/`free`, а не embind — чтобы контролировать
  границу самим.
- Дисциплина памяти: не кэшировать `Module.HEAPF32` (протухает при росте кучи),
  каждый `_malloc` парить с `_free` через `try/finally`, результат копировать из
  кучи через `.slice()`.
- Через worker буферы ходят transferable (нулевое копирование на уровне
  `postMessage`; копии остаются только внутри wasm-обёртки).
