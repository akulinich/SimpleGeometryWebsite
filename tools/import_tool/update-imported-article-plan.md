# План: обновление уже импортированной статьи

## Цель

Добавить в `tools/import_tool` отдельный сценарий обновления статьи, которая уже была импортирована на сайт.

Обновление должно поддерживать два типа изменений:

- изменился markdown-текст исходной статьи в Obsidian;
- изменился набор картинок или содержимое картинок.

Текущий импорт должен остаться сценарием для новых статей. Обновление должно работать только для статей со статусом `imported`.

## Текущее поведение

Сейчас `ImportTool` строит список через `get_all_articles()`:

- `new` - есть в Obsidian, но нет `src/content/articles/ru/{id}.md`;
- `imported` - есть в Obsidian и есть `src/content/articles/ru/{id}.md`;
- `site_only` - есть на сайте, но исходника в Obsidian нет.

Импорт запускает `PipelineDialog` для выбранного файла. Внутри пайплайна:

- шаги 1-4 могут менять исходную заметку в Obsidian;
- шаг 8 создает или перезаписывает `src/content/articles/en/{id}.md`;
- шаг 10 копирует исходную заметку в `src/content/articles/ru/{id}.md` и копирует картинки в `public/images`;
- шаг 11 проставляет ссылки в `ru` и `en`.

## UX

Добавить рядом с кнопкой `Импортировать выбранную` кнопку:

```text
Обновить выбранную
```

Правила:

- выбрана ровно одна статья;
- статья должна иметь статус `imported`;
- для `new` показать сообщение: сначала используйте импорт;
- для `site_only` показать сообщение: нет исходника в Obsidian, обновлять нечего.

Перед запуском обновления показать диалог подтверждения:

```text
Обновить статью "{id}" на сайте?

Будут обновлены:
- русская версия;
- английская версия, если выбран шаг перевода и русский текст изменился;
- картинки, которые использует статья.

Старые неиспользуемые картинки будут удалены автоматически, если они больше не используются на сайте.
```

В диалоге пайплайна для update-сценария должны быть другие дефолты шагов:

- включить анализ текста опционально, но по умолчанию выключить;
- включить анализ картинок опционально, но по умолчанию выключить;
- включить перевод по умолчанию, но пропускать шаг автоматически, если русский markdown не изменился;
- включить проверку терминологии по умолчанию;
- включить копирование файлов по умолчанию;
- включить проставление ссылок по умолчанию;
- выключить заполнение `id`, потому что у импортированной статьи id уже должен быть;
- заполнение `title` оставить опциональным.

## Архитектура

Ввести режим пайплайна:

```python
mode: Literal['import', 'update']
```

Передавать его в `PipelineDialog`.

Для текущего импорта использовать `mode='import'`.
Для новой кнопки обновления использовать `mode='update'`.

В `PipelineDialog` режим нужен для:

- набора шагов по умолчанию;
- текстов окна;
- поведения шага копирования;
- поведения шага перевода;
- финального сообщения.

## Поиск статьи

Для обновления использовать уже существующую связку:

- выбранный элемент списка дает `display_path`, `status`, `article_id`;
- `article_id` соответствует файлам:
  - `src/content/articles/ru/{article_id}.md`;
  - `src/content/articles/en/{article_id}.md`.

Если `article_id` пустой, обновление запрещать и просить заполнить `id` в исходной заметке.

## Обновление русского markdown

В update-сценарии копирование русской версии должно:

1. прочитать исходник из Obsidian;
2. прочитать текущий `ru/{id}.md`, если он есть;
3. показать diff или хотя бы короткое предупреждение, если файл будет перезаписан;
4. перезаписать `ru/{id}.md` исходником из Obsidian;
5. сохранить `utf-8` и `newline='\n'`.

Минимальная первая версия может не показывать diff, но должна явно писать в лог пайплайна:

```text
ru/{id}.md обновлен
```

## Обновление английского markdown

Для update-сценария сразу делать улучшенную версию, без полного повторного перевода по умолчанию.

Добавить отдельный prompt:

```text
tools/import_tool/prompts/update_translation.md
```

Prompt получает три блока:

- старый русский `src/content/articles/ru/{id}.md`;
- новый русский markdown из Obsidian;
- текущий английский `src/content/articles/en/{id}.md`.

Задача модели:

- сравнить старый и новый русский текст;
- обновить английскую версию только там, где русский текст изменился;
- сохранить стиль, структуру, frontmatter, markdown-разметку и терминологию текущей английской версии;
- не переписывать неизмененные абзацы без необходимости;
- вернуть полный итоговый `en/{id}.md`, а не patch.

Новый update-translation шаг должен работать так:

1. До обновления `ru/{id}.md` сохранить старый русский markdown в `self._old_ru_content`.
2. Прочитать новый русский markdown из Obsidian в `self._new_ru_content`.
3. Если `self._old_ru_content == self._new_ru_content`, пропустить шаг без вызова Claude и записать в лог, что русский текст не изменился.
4. Прочитать текущий английский `en/{id}.md`.
5. Отправить все три блока в Claude через `update_translation.md`.
6. Показать результат в text area.
7. По кнопке `Применить` перезаписать `en/{id}.md`.
8. После этого продолжить пайплайн к проверке терминологии и проставлению ссылок.

Если `en/{id}.md` отсутствует, использовать fallback:

- показать предупреждение в логе пайплайна;
- выполнить обычный `translate.md` по новому русскому markdown;
- сохранить результат как новую английскую версию.

В коде лучше не смешивать обычный импортный перевод и update-перевод в одном большом методе. Разделить:

```python
def _step_translate(self):
    if self.mode == 'update':
        self._step_update_translation()
    else:
        self._step_full_translation()
```

Для контроля токенов prompt должен явно запрещать полный пересказ английской версии, если русские изменения локальные. Но результат все равно должен быть полным файлом, чтобы сохранение было простым и безопасным.

## Обновление title

`title` используется сайтом как пользовательский заголовок статьи:

- в `<h1>` на странице статьи;
- в `<title>` HTML-страницы через `BaseLayout`;
- в списке статей;
- в поиске по статьям, потому что индекс фильтра строится из `article.data.title`.

Если `title` отсутствует, сайт использует slug как fallback, но это хуже для читателя и поиска.

Для `ru/{id}.md` title приходит из исходной Obsidian-заметки вместе с frontmatter, потому что русский markdown копируется целиком.

Для `en/{id}.md` title должен обновляться внутри `update_translation.md`, если русский `title` изменился. Prompt должен явно сказать:

- `id`, `date`, `draft`, `references` не менять;
- `title`, `description`, `tags` обновлять только если изменились соответствующие русские поля;
- если русский `title` не изменился, сохранить текущий английский `title`.

## Обновление картинок

Нужно учитывать два независимых случая:

- в markdown изменился список ссылок на картинки;
- файл картинки с тем же именем изменился в папке Obsidian images.

Алгоритм:

1. До копирования прочитать старый `ru/{id}.md` и получить `old_images = find_images_in_md(old_content)`.
2. Прочитать новый исходный markdown и получить `new_images = find_images_in_md(new_content)`.
3. Для каждой картинки из `new_images`:
   - найти файл в `images_folder`;
   - если файла нет, записать warning в лог;
   - если файл есть, скопировать в `public/images`, перезаписывая существующий файл.
4. Посчитать `removed_images = old_images - new_images`.
5. Для каждой картинки из `removed_images` проверить, используется ли она где-либо на сайте.
6. Если картинка больше нигде не используется, автоматически удалить ее из `public/images`.
7. Записать в лог список перезалитых, отсутствующих и удаленных картинок.

Проверку использования можно вынести в общий helper:

```python
def find_images_used_by_site(exclude_article_id=None) -> set[str]:
    ...
```

Он должен читать все `src/content/articles/ru/*.md` и `src/content/articles/en/*.md`.

Важно: английская версия может ссылаться на те же картинки. Поэтому удаление картинки безопасно только если она не найдена ни в одном текущем `ru` или `en` markdown-файле сайта.

## Конфликты и безопасность

Нельзя молча менять `id` импортированной статьи.

Если в исходной заметке `id` отличается от выбранного `article_id`:

- остановить обновление;
- показать ошибку;
- предложить сначала руками разобраться с id.

Если `ru/{id}.md` отсутствует, хотя статус `imported`:

- показать ошибку состояния;
- предложить обновить список или выполнить обычный импорт.

Если `en/{id}.md` отсутствует:

- разрешить update;
- шаг перевода должен создать английскую версию.

## Тест-план

Ручные проверки:

1. Выбрать `new` статью и нажать `Обновить выбранную` - должно быть предупреждение.
2. Выбрать `site_only` статью и нажать `Обновить выбранную` - должно быть предупреждение.
3. Выбрать `imported` статью, изменить текст в Obsidian, выполнить update без перевода - обновляется только `ru/{id}.md`.
4. Выполнить update с переводом - обновляются `ru/{id}.md` и `en/{id}.md`.
5. Добавить новую картинку в markdown - файл копируется в `public/images`.
6. Изменить файл картинки с тем же именем - файл в `public/images` перезаписывается.
7. Удалить картинку из markdown - если она нигде больше не используется, инструмент автоматически удаляет ее.
8. Удалить картинку из markdown, но оставить ее в другой статье или текущей английской версии - инструмент не удаляет файл.
9. Поменять `id` в исходнике - update останавливается с ошибкой.

Автоматические проверки:

- `python -m py_compile tools/import_tool/import_tool.py tools/import_tool/core.py`;
- unit-тесты для `find_images_used_by_site()` на временной структуре файлов;
- unit-тесты для расчета `old_images/new_images/removed_images`.
